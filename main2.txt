2. Служба DNS. Компоненты DNS. Протоколы DNS.

DNS (Domain Name System) — распределённая система для преобразования имён доменов в IP-адреса и наоборот.

Компоненты DNS:

DNS-резолвер (клиент) — программный модуль на устройстве, формирующий запрос.

Рекурсивный резолвер — сервер (например, провайдера), который получает запрос от клиента и выполняет весь процесс разрешения имени.

Корневые серверы (root-servers) — указывают на серверы доменов верхнего уровня.

Серверы доменов верхнего уровня (TLD-servers) — хранят информацию о доменах следующего уровня (.com, .ru, .org).

Авторитативные серверы (authoritative name servers) — конечный источник истины для конкретного домена, хранят его DNS-записи.

 

Протоколы DNS:

UDP/53 — основной протокол для стандартных запросов (быстрый, с малым заголовком).

TCP/53 — используется, когда ответ не помещается в один UDP-пакет (>512 байт), а также для передачи зон (zone transfer — AXFR/IXFR).

(DoH)— отправка DNS-запросов в формате HTTPS, маскируя их под обычный веб-трафик.

4. СУБД. MySQL. Особенности, архитектурные компоненты, область применения.

 

MySQL —реляционная СУБД с открытым исходным кодом

Особенности MySQL:

Разделение задач: Сервер обрабатывает запросы, а движки (InnoDB, MyISAM) отвечают за хранение. Можно менять движки под задачу.

Много движков: InnoDB — для надёжности и транзакций, MyISAM — для быстрого чтения.

Репликация: Автоматическое копирование данных на резервные серверы для отказоустойчивости.

Кроссплатформенность: Работает на Windows, Linux, macOS.

Многопользовательский доступ: Много людей могут работать одновременно с разными правами доступа.

 

Архитектурные компоненты:

Клиент: Программа (сайт, приложение), которая отправляет SQL-запросы.

Сервер (ядро MySQL):

SQL-интерфейс: Принимает и проверяет запросы.

Оптимизатор: Ищет самый быстрый способ выполнить запрос.

Кэш запросов: Хранит готовые ответы для ускорения.

Диспетчер потоков: Управляет одновременными запросами.

Уровень хранилища: Работает с выбранным движком (InnoDB/MyISAM) через единый API.

Область применения:

Веб-сайты и приложения: Динамические сайты, интернет-магазины (основа для WordPress, Joomla).

Бизнес-системы: ERP, CRM для управления данными компаний.

Аналитика: Хранилище данных для отчётов (чаще для средних объёмов).

Мобильные приложения: Хранение пользовательских данных.

Встроенные системы: Используется в CMS, блогах, сетевом оборудовании.

 

5. Веб-сервер. Особенности. Типы. Компоненты

 

Веб-сервер — это программное обеспечение, принимающее HTTP/HTTPS-запросы от клиентов (браузеров) и возвращающее им веб-страницы, файлы или данные.

Особенности веб-сервера:
Обработка входящих запросов — сервер принимает запросы, разбирает их и ищет запрошенный ресурс


Работа с протоколами — использует стандартные протоколы HTTP и HTTPS для обмена данными

 


Масштабируемость — поддерживает множество одновременных подключений пользователей


Журналирование и безопасность — ведёт логи, контролирует доступ и отслеживает подозрительную активность


Поддержка динамического контента — может взаимодействовать с серверными языками (PHP, Python, Java) для генерации страниц "на лету"


Работа со статическим контентом — эффективно отдаёт готовые файлы (HTML, CSS, изображения)

Типы веб-серверов:
Apache HTTP Server — гибкий и расширяемый, с поддержкой различных модулей


Nginx — высокопроизводительный, эффективно справляется с большим количеством одновременных подключений, часто используется как обратный прокси


Microsoft IIS — веб-сервер от Microsoft, интегрированный с Windows Server
Статические серверы — отдают только готовые файлы без выполнения серверной логики


Динамические серверы — выполняют серверный код и генерируют контент в зависимости от запроса

Компоненты веб-сервера:
HTTP-сервер — принимает и обрабатывает входящие HTTP-запросы


Файловая система — хранит веб-контент (HTML-файлы, изображения, скрипты)


Программное обеспечение веб-сервера — ядро, отвечающее за обработку запросов и формирование ответов


Интеграция с сервером приложений — взаимодействие с дополнительным ПО для выполнения бизнес-логики (при необходимости)

12. Аутентификация через ACL-листы. Отличие ACL от стандартных

методов. Принцип работы ACL-листа.

ACL (Access Control List) — это список контроля доступа, который определяет, кто и какие действия может выполнять с определённым ресурсом. Он работает как фильтр, разрешая или запрещая доступ на основе заранее заданных правил.

Отличие ACL от стандартных методов — гибкость: можно задавать нюансированные уровни доступа для различных участников. Например, один сотрудник может иметь право на чтение документа, в то время как другой — на его изменение.

Принцип работы ACL: пакеты проверяются на соответствие каждому правилу в списке, пока не будет найдено совпадение. Если совпадение не найдено, применяется действие по умолчанию. Каждое правило обычно содержит три ключевых элемента: субъект, объект и тип разрешённого или запрещённого действия

15. Службы каталога. Протокол LDAP. Компоненты протокола

LDAP (Lightweight Directory Access Protocol) — протокол для доступа и управления распределёнными каталогами информации. Позволяет централизованно хранить и быстро получать данные о пользователях, группах, устройствах, сервисах и других объектах.

Некоторые компоненты протокола LDAP:

• LDAP-сервер — хранит и обслуживает каталог.
• LDAP-клиенты — программы, которые обращаются к серверу.
• Схемы (schemas) — определяют типы объектов и атрибутов.
• ACL (Access Control List) — правила доступа к данным.

17. Обзор механизмов управления политиками в Linux (PAM, sudo,

ACL, SELinux, AppArmor)

PAM обеспечивает динамическое управление аутентификацией в системах Linux.

PAM является посредником между приложениями (SSH, sudo и др.) и методами аутентификации. При попытке входа приложение передаёт PAM запрос на аутентификацию, а PAM обрабатывает его согласно правилам и модулям, указанным в своих конфигурационных файлах.

SUDO

Позволяет пользователям выполнять команды с привилегиями суперпользователя или учётной записи root

ACL (Access Control Lists)

Позволяет более гибко задавать права для конкретных пользователей и групп. Каждому объекту в системе соответствует свой список доступа. В нём прописаны все пользователи, у которых есть права на этот объект, и указано, что они могут делать. 

SELinux

Реализует систему принудительного контроля доступа. Права доступа определяются самой системой при помощи специально определённых политик.

SELinux работает в трёх основных режимах:

• Enforcing —доступы, запрещённые политикой, блокируются и регистрируются. Это режим по умолчанию.
• Permissive - доступы, запрещённые политикой, регистрируются, но не блокируются. Этот режим полезен для устранения неполадок и тестирования новых политик.
• Disabled — SELinux отключён, политика не загружена и не применяется. Этот режим обычно не рекомендуется для производственных систем.

• Поддержка PHP, популярного языка веб-разработки.
• Возможность работать в разных операционных системах и поддерживать множество языков программирования.

22. Службы каталога. Протокол LDAP. Репликация протокола.

LDAP (Lightweight Directory Access Protocol) — протокол для доступа и управления распределёнными каталогами информации. Позволяет централизованно хранить и быстро получать данные о пользователях, группах, устройствах, сервисах и других объектах.

1. Репликация в LDAP — это процесс синхронизации данных между несколькими серверами каталогов, чтобы они содержали одинаковую информацию. Когда данные изменяются на главном сервере, эти изменения автоматически копируются на подчинённые серверы.
Два вида:

• Master-Slave: Все новые записи делаются только в главную тетрадь, а потом переписываются в копии.
• Multi-Master: Есть несколько главных тетрадей, и записи из любой из них переписываются во все остальные.

3. Облачные (SaaS)

Пример: Datadog — устанавливаете агент на сервер, он отправляет метрики в облако Datadog. В веб-интерфейсе видите графики, логи, трассировки. Настраиваете алерт при высокой задержке БД.

39. СУБД. Сравнение MongoDB, MariaDB и MySQL.

Краткая характеристика:

• MariaDB — форк MySQL с улучшениями (open source)
• MongoDB — документо-ориентированная NoSQL СУБД
Когда что выбирать:

Выбирайте MySQL, если:

• Требуется больше оптимизаций "из коробки"
• Опасаетесь влияния Oracle на MySQL
Выбирайте MongoDB, если:

• Работаете с неструктурированными/полуструктурированными данными
• Нужна быстрая разработка без схемы
• Требуется горизонтальное масштабирование
• Большие объёмы данных, которые растут быстро
• Нет сложных связей между сущностями

43. Политики безопасности в Linux. Принципы контроля доступа.

Методы контроля доступа.

Контроль доступа — это набор правил, определяющих, кому, к каким ресурсам и какие действия разрешены.

 
Основные принципы:

1. Принцип наименьших привилегий (POLP)
o Пользователь получает только те права, которые необходимы для его работы
o Пример: Пользователю для работы с файлами не нужен доступ sudo
2. Разделение обязанностей (SoD)
o Критичные операции требуют участия нескольких лиц
o Пример: Настройка firewall + мониторинг логов разными администраторами
3. Обязательный контроль доступа (MAC)
o Правила задаются системой, а не владельцем ресурса
o Пример: SELinux политики
4. Дискреционный контроль доступа (DAC)
o Владелец ресурса сам решает, кому давать доступ
o Пример: Права rwx в Linux (chmod, chown)
5. Ролевой контроль доступа (RBAC)
o Права выдаются на основе ролей пользователя
o Пример: Роли "администратор", "оператор", "аналитик"
Методы

• Избирательное (дискреционное) управление доступом (Discretionary Access Control, DAC). Каждому объекту системы назначается список пользователей, для каждого из них задаётся список допустимых операций (читать, писать и т. д.). Каждый объект имеет привязанного к нему пользователя — владельца, который устанавливает права доступа. 
• Мандатное управление доступом (Mandatory Access Control, MAC). Позволяет задать явные правила доступа субъектов (программ, действующих от имени пользователя) к объектам (файлы, устройства, сокеты, порты, процессы) в виде политики доступа. Политика задаётся не владельцем объекта и не может быть изменена. Права на передачу прав определяются также в политике. 
• Контроль доступа на основе ролей (Role-Based Access Control, RBAC). Права доступа реализованы в виде ролей, выдаваемых системой безопасности. Роли — это полномочия на выполнение конкретных действий группой субъектов над объектами в системе. 
 

 

44. Домен. Компоненты. Принцип работы. Службы каталога

Домен — это логическая группа компьютеров и пользователей, управляемая как единое целое с общей базой данных (каталогом), политиками безопасности и централизованным управлением.

Основные компоненты домена:

1. Контроллер домена (DC) — главный сервер, хранящий базу данных пользователей, компьютеров и политик.
2. База данных каталога — централизованное хранилище учётных записей (обычно на основе LDAP).
3. Клиенты домена — компьютеры и пользователи, присоединённые к домену.
4. Доверительные отношения — связи между разными доменами для совместного доступа.
Принцип работы домена:

1. Присоединение компьютера к домену:

text

Компьютер → Запрос к контроллеру домена → Проверка прав администратора

          → Запись в базу каталога → Настройка политик → Готово

2. Вход пользователя:

text

Пользователь вводит логин/пароль → Запрос к DC → Проверка в каталоге

→ Если верно → DC возвращает токен доступа → Пользователь получает доступ

→ Одновременно применяются групповые политики

3. Централизованное управление:

• Единая авторизация — один логин/пароль для всех ресурсов домена
• Групповые политики (GPO) — настройки для пользователей и компьютеров
• Делегирование прав — администраторы могут давать права по отделам
Службы каталога

Служба каталога — это сетевая служба, которая:

• Хранит информацию о пользователях, компьютерах, принтерах
• Обеспечивает аутентификацию и авторизацию
• Управляет политиками доступа
 

45. Политики безопасности в Linux. Аутентификация, авторизация,

Учет

1. Аутентификация: Ты вводишь пароль в Linux → система проверяет, правильный ли он.
2. Авторизация: Если пароль верный, система смотрит, можно ли тебе, например, устанавливать программы.
3. Учёт: Система записывает в лог, что ты установил программу nginx в 15:00.
Итог:
Сначала система проверяет кто ты (аутентификация), потом решает что тебе можно (авторизация), и записывает что ты сделал (учёт).

 

46. Виртуализация. Сферы применения. Основные типы

виртуализации.

Виртуализация — создание виртуальных (не физических) версий серверов, компьютеров, сетей или хранилищ на одном физическом оборудовании.

 
Сферы применения:

1. Серверная консолидация — несколько серверов на одном железе.
2. Разработка и тестирование — изолированные среды для приложений.
3. Виртуальные рабочие столы (VDI) — доступ к рабочему столу с любого устройства.
4. Облачные платформы — основа IaaS (AWS, Azure).
5. Контейнеризация — изоляция приложений (Docker, Kubernetes).
 
Основные типы виртуализации:

1. Аппаратная (полная)

• Виртуализируется всё железо (CPU, RAM, диски).
• Примеры: VMware ESXi, Microsoft Hyper-V, KVM.
2. Программная (эмуляция)

• Виртуализация через программный слой поверх ОС.
• Примеры: VirtualBox, VMware Workstation.
3. Контейнерная

• Виртуализация на уровне ОС: общее ядро, изолированные процессы.
• Примеры: Docker, LXC.
4. По уровню гипервизора:

• Type 1 (bare-metal) — гипервизор на голом железе (ESXi, Hyper-V).
• Type 2 (hosted) — гипервизор внутри ОС (VirtualBox).
5. По объекту:

• Сети — виртуальные коммутаторы и маршрутизаторы.
• Хранилища — виртуальные дисковые массивы.
• Приложений — изоляция без виртуализации ОС.
47. Служба DNS. Иерархическая структура DNS.

Иерархическая структура DNS

DNS имеет древовидную иерархическую структуру, похожую на перевёрнутое дерево или организационную схему компании.

 
Уровни иерархии (сверху вниз):

1. Корневой уровень (root) — верхушка иерархии
o Обозначается точкой . (часто опускается)
o 13 групп корневых серверов (буквенно: a.root-servers.net — m.root-servers.net)
o Хранят информацию о серверах доменов верхнего уровня
2. Домены верхнего уровня (TLD — Top-Level Domains)
o gTLD (общие): .com, .org, .net, .info
o ccTLD (национальные): .ru, .de, .us, .cn
o Специальные: .gov, .edu, .mil
3. Домены второго уровня (Second-Level Domains)
o Имена, которые регистрируют пользователи
o Примеры: google в google.com, yandex в yandex.ru
4. Субдомены (поддомены, Subdomains)
o Дополнительные уровни перед основным доменом
o Примеры: mail.google.com, docs.google.com
